## 2. 学习顺序完成了一次重要校准

最终主线确定为：

> Python 后端基础 -> 数据结构与算法 -> 完整后端项目 -> RAG -> Agent -> 部署与实习面试

这意味着：

- 不直接跳到 Agent 框架，先补 Python 工程与后端基础。
- 数据结构与算法需要学，但以开发和常见面试所需为边界，不走竞赛路线。
- 先做可用的 RAG 系统，再把它升级成 Agent 应用。
- Agent 框架先专注 LangGraph，避免同时铺开多个框架。
- 不以“看完课程”为完成标准，而以独立编码、测试、讲解和验收为准。

## 3. 第一阶段目标已经明确

2026 年 7 月至 8 月底需要完成：

1. 盘点并补齐 Python 进阶知识，包括模块、异常、文件与 JSON、类型标注、虚拟环境、API 调用、异步、测试和日志。
2. 掌握 Git / GitHub 与 Linux 基本工作流，后续项目保留真实提交记录。
3. 用有限时间学习 C 指针、内存和基础数据结构，帮助理解底层机制。
4. 完成约 20 至 30 道覆盖核心题型的简单算法题，并维护错题记录。
5. 完成一个 Python 命令行小项目，验收模块化、外部 API、异常处理、日志和测试。

## 4. 项目路线从零散想法变成连续成长路径

- 项目零：Python 命令行小项目，验证基础工程能力。
- 项目一：用户管理和资料管理后端，加入数据库、认证、权限、测试和部署。
- 项目二：在完整后端上开发知识库 RAG 系统，加入权限、引用、历史记录和日志。
- 项目三：不推倒重来，在 RAG 系统上加入工具、状态、人工确认和失败恢复，升级为 Agent 应用。

这条路线的价值在于，每个项目都复用前一阶段的能力，最终作品集能展示持续迭代，而不是几个互不相关的演示程序。

## 5. 我对学习方式有了更清楚的要求

- 项目学习时间显著多于算法训练时间。
- 每次只推进一个可验收的小任务。
- 核心模块至少独立重写一次，避免只会在辅助工具提示下完成。
- 每次学习都留下 GitHub 可见成果：代码、测试、README、运行记录或复盘。
- Code Review 不只检查能否运行，也检查命名、结构、边界情况、错误处理和可维护性。
