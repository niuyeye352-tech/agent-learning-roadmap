## 能力变化

### 1. 从“语法练习”转向“明确的数据流”

现在能够说明状态从哪里进入函数、在哪里被处理、通过什么返回，并由调用处决定是否更新外部变量。闭包练习进一步展示了状态如何在多次调用之间保留，也理解了 `nonlocal` 的适用边界。

### 2. 从散落变量转向对象建模

类、对象、继承和封装被放在 Agent 任务场景中练习：公共属性和行为放在父类，不同 Agent 通过子类扩展；内部状态由公开接口读取和校验修改。这里最重要的收获不是会写类，而是开始形成“数据与行为由对象自己管理”的职责意识。

### 3. 从单文件转向多文件协作

Day 16 已经能够把可复用的类放入模型模块，把函数放入工具模块，再由主程序导入和调度。也能够使用主程序入口保护，避免模块被导入时意外执行演示或业务代码。这是从基础练习过渡到项目结构的重要一步。

### 4. 形成了更严格的验收习惯

每个学习日都不只检查“能否运行”，还会核对输出、状态变化、变量命名、类型标注、空行格式和导入副作用。Day 16 的最终验收还检查了模块直接运行与被导入时的不同行为，并完成语法编译检查。

## 与 Agent 应用开发的连接

- Agent 的名称、状态和行为可以由类与实例方法表达。
- 不同类型的 Agent 可以通过继承和方法重写共享公共能力并保留差异。
- `property` 和 setter 可以作为简单状态规则的统一入口。
- 工具函数、Agent 模型和主程序入口可以拆分到不同模块，减少复制并明确职责。
- 主程序入口保护能让模块既可复用，也可独立运行和验证。

这些练习还不是真正的 Agent 系统，但已经在为后续的工具调用、状态管理和多模块工程结构建立 Python 基础。
